# Qlib Backtest Indicator Update Flow

This note explains the full flow of `Account.update_indicator()` and how `Indicator` updates order-level and step-level trading metrics.

The key point is that `Indicator` has two execution paths:

```text
atomic=True
    The current executor is the lowest-level executor.
    It directly receives real trade results from `trade_info`.

atomic=False
    The current executor is an outer nested executor.
    It does not trade directly.
    It aggregates multiple inner step indicators into one outer step indicator.
```

---

## 1. Entry Point: `Account.update_indicator()`

The entry function is:

```python
def update_indicator(
    self,
    trade_start_time: pd.Timestamp,
    trade_exchange: Exchange,
    atomic: bool,
    outer_trade_decision: BaseTradeDecision,
    trade_info: list = [],
    inner_order_indicators: List[BaseOrderIndicator] = [],
    decision_list: List[Tuple[BaseTradeDecision, pd.Timestamp, pd.Timestamp]] = [],
    indicator_config: dict = {},
) -> None:
    """update trade indicators and order indicators in each bar end"""

    self.indicator.reset()

    if atomic:
        self.indicator.update_order_indicators(trade_info)
    else:
        self.indicator.agg_order_indicators(
            inner_order_indicators,
            decision_list=decision_list,
            outer_trade_decision=outer_trade_decision,
            trade_exchange=trade_exchange,
            indicator_config=indicator_config,
        )

    self.indicator.cal_trade_indicators(trade_start_time, self.freq, indicator_config)

    self.indicator.record(trade_start_time)
```

The process is:

```text
1. reset()
2. build order-level indicators
3. aggregate order-level indicators into step-level indicators
4. record the result into history
```

---

## 2. What `Indicator` Stores

`Indicator` stores two kinds of metrics.

### 2.1 `order_indicator`

`order_indicator` is stock-level.

Each row is one stock in the current step.

Example:

```text
              amount   inner_amount   deal_amount   trade_price   base_price   ffr     pa
SH600000      1200     1000           1000          10.10         10.15        0.83    0.0049
SH600001      -500     -500           -300          19.90         19.90        0.60    0.0000
```

Main fields:

```text
amount
    Target amount from the outer strategy.
    Buy is positive. Sell is negative.

inner_amount
    Total target amount generated by the inner strategy.
    Buy is positive. Sell is negative.

deal_amount
    Real executed amount.
    Buy is positive. Sell is negative.

trade_price
    Average executed price.

trade_value
    Real executed value.
    Buy is positive. Sell is negative.

trade_cost
    Total trading cost.

trade_dir
    Trading direction.
    BUY = 1, SELL = 0.

ffr
    Full fill rate.
    ffr = deal_amount / amount.

base_price
    Baseline market price used to calculate price advantage.

base_volume
    Weight used to aggregate base_price.
    In TWAP mode, it is usually the number of valid price points.
    In VWAP mode, it is usually market volume.

pa
    Price advantage.
    For buy orders, lower trade_price than base_price is positive.
    For sell orders, higher trade_price than base_price is positive.
```

---

### 2.2 `trade_indicator`

`trade_indicator` is step-level.

It aggregates all stock-level `order_indicator` values in the current step.

Main fields:

```text
ffr
    Aggregated full fill rate.

pa
    Aggregated price advantage.

pos
    Positive rate.
    It is the ratio of orders whose pa > 0.

deal_amount
    Total executed amount, using absolute values.

value
    Total executed value, using absolute values.

count
    Number of outer orders in the current step.
```

---

## 3. High-Level Flow

The high-level flow is:

```text
Account.update_indicator()
    ↓
Indicator.reset()
    ↓
if atomic=True:
    Indicator.update_order_indicators(trade_info)
else:
    Indicator.agg_order_indicators(...)
    ↓
Indicator.cal_trade_indicators(...)
    ↓
Indicator.record(trade_start_time)
```

---

# 4. Path 1: atomic=True

`atomic=True` means the current executor is the lowest-level executor.

It directly receives real trade results.

Input:

```text
trade_info
```

Typical structure:

```text
[
    (order, trade_value, trade_cost, trade_price),
    ...
]
```

The flow is:

```text
update_order_indicators(trade_info)
    ↓
_update_order_trade_info(trade_info)
    ↓
_update_order_fulfill_rate()
```

---

## 4.1 `_update_order_trade_info()`

This function converts real trade results into stock-level metrics.

Assume the current lowest-level executor has two orders:

```text
Order 1:
    Buy SH600000
    target amount = 1000
    executed amount = 1000
    trade_price = 10.00
    trade_value = 10000
    trade_cost = 5

Order 2:
    Sell SH600001
    target amount = 500
    executed amount = 300
    trade_price = 20.00
    trade_value = 6000
    trade_cost = 5
```

Because buy is positive and sell is negative:

```text
amount:
SH600000      1000
SH600001      -500

inner_amount:
SH600000      1000
SH600001      -500

deal_amount:
SH600000      1000
SH600001      -300

trade_price:
SH600000      10.00
SH600001      20.00

trade_value:
SH600000      10000
SH600001      -6000

trade_cost:
SH600000      5
SH600001      5

trade_dir:
SH600000      BUY
SH600001      SELL

pa:
SH600000      0
SH600001      0
```

In the lowest-level executor:

```text
amount = inner_amount
```

because there is no lower executor inside it.

The `pa` value is set to `0` in this step because price advantage is meaningful only when comparing against an aggregated baseline price.

---

## 4.2 `_update_order_fulfill_rate()`

This function calculates:

```text
ffr = deal_amount / amount
```

Example:

```text
amount:
SH600000      1000
SH600001      -500

deal_amount:
SH600000      1000
SH600001      -300
```

Result:

```text
ffr:
SH600000      1000 / 1000 = 1.0
SH600001      -300 / -500 = 0.6
```

So the final lowest-level `order_indicator` is:

```text
              amount   inner_amount   deal_amount   trade_price   trade_value   trade_cost   trade_dir   ffr   pa
SH600000      1000     1000           1000          10.00         10000         5            BUY         1.0   0
SH600001      -500     -500           -300          20.00         -6000         5            SELL        0.6   0
```

---

# 5. Path 2: atomic=False

`atomic=False` means the current executor is an outer nested executor.

Example:

```text
Outer step = one day
Inner step = one minute
```

The outer executor does not trade directly.
The real trades happen in the inner minute executor.

Therefore, the outer `Indicator` must aggregate several inner `order_indicator` objects into one outer `order_indicator`.

The entry function is:

```python
agg_order_indicators(...)
```

The internal order is:

```python
self._agg_order_trade_info(inner_order_indicators)
self._update_trade_amount(outer_trade_decision)
self._update_order_fulfill_rate()
pa_config = indicator_config.get("pa_config", {})
self._agg_base_price(inner_order_indicators, decision_list, trade_exchange, pa_config=pa_config)
self._agg_order_price_advantage()
```

Dependency graph:

```text
inner_order_indicators
    ↓
_agg_order_trade_info()
    produces:
        inner_amount
        deal_amount
        trade_price
        trade_value
        trade_cost
        trade_dir

outer_trade_decision
    ↓
_update_trade_amount()
    produces:
        amount

amount + deal_amount
    ↓
_update_order_fulfill_rate()
    produces:
        ffr

inner_order_indicators + decision_list + trade_exchange + trade_dir
    ↓
_agg_base_price()
    produces:
        base_price
        base_volume

trade_price + base_price + trade_dir
    ↓
_agg_order_price_advantage()
    produces:
        pa
```

---

## 5.1 `_agg_order_trade_info()`

### Purpose

This function aggregates real execution information from inner steps.

It only uses:

```text
inner_order_indicators
```

It reads these metrics from each inner step:

```text
inner_amount
deal_amount
trade_price
trade_value
trade_cost
trade_dir
```

It produces the outer-level:

```text
inner_amount
deal_amount
trade_price
trade_value
trade_cost
trade_dir
```

It does not produce `amount`.

`amount` comes from the outer original decision and is added later by `_update_trade_amount()`.

---

### Example Input

Assume one outer day contains three inner minute steps:

```text
step0 = 09:30
step1 = 09:31
step2 = 09:32
```

The outer day involves two stocks:

```text
SH600000: buy
SH600001: sell
```

Inner step data:

### step0

```text
inner_amount:
SH600000      300
SH600001     -200

deal_amount:
SH600000      300
SH600001     -100

trade_price:
SH600000      10.00
SH600001      20.00

trade_value:
SH600000      3000
SH600001     -2000

trade_cost:
SH600000      2
SH600001      2

trade_dir:
SH600000      BUY
SH600001      SELL
```

### step1

```text
inner_amount:
SH600000      400
SH600001     -200

deal_amount:
SH600000      400
SH600001     -100

trade_price:
SH600000      10.10
SH600001      19.90

trade_value:
SH600000      4040
SH600001     -1990

trade_cost:
SH600000      2
SH600001      2

trade_dir:
SH600000      BUY
SH600001      SELL
```

### step2

```text
inner_amount:
SH600000      300
SH600001     -100

deal_amount:
SH600000      300
SH600001     -100

trade_price:
SH600000      10.20
SH600001      19.80

trade_value:
SH600000      3060
SH600001     -1980

trade_cost:
SH600000      2
SH600001      2

trade_dir:
SH600000      BUY
SH600001      SELL
```

---

### Aggregating `inner_amount`

```text
SH600000:
300 + 400 + 300 = 1000

SH600001:
-200 + -200 + -100 = -500
```

Result:

```text
inner_amount:
SH600000      1000
SH600001      -500
```

---

### Aggregating `deal_amount`

```text
SH600000:
300 + 400 + 300 = 1000

SH600001:
-100 + -100 + -100 = -300
```

Result:

```text
deal_amount:
SH600000      1000
SH600001      -300
```

---

### Aggregating `trade_price`

`trade_price` is not averaged directly.

First, each inner step transforms:

```text
trade_price
```

into:

```text
deal_amount * trade_price
```

This is the numerator of the weighted average trade price.

For `SH600000`:

```text
step0: 300 * 10.00 = 3000
step1: 400 * 10.10 = 4040
step2: 300 * 10.20 = 3060
```

Then:

```text
weighted numerator = 3000 + 4040 + 3060 = 10100
deal_amount sum    = 300 + 400 + 300 = 1000

trade_price = 10100 / 1000 = 10.10
```

For `SH600001`, sell amounts are negative:

```text
step0: -100 * 20.00 = -2000
step1: -100 * 19.90 = -1990
step2: -100 * 19.80 = -1980
```

Then:

```text
weighted numerator = -2000 + -1990 + -1980 = -5970
deal_amount sum    = -100 + -100 + -100 = -300

trade_price = -5970 / -300 = 19.90
```

Result:

```text
trade_price:
SH600000      10.10
SH600001      19.90
```

---

### Aggregating `trade_value`

```text
SH600000:
3000 + 4040 + 3060 = 10100

SH600001:
-2000 + -1990 + -1980 = -5970
```

Result:

```text
trade_value:
SH600000      10100
SH600001      -5970
```

---

### Aggregating `trade_cost`

```text
SH600000:
2 + 2 + 2 = 6

SH600001:
2 + 2 + 2 = 6
```

Result:

```text
trade_cost:
SH600000      6
SH600001      6
```

---

### Aggregating `trade_dir`

For buy orders:

```text
BUY = 1
```

For sell orders:

```text
SELL = 0
```

During aggregation:

```text
SH600000:
1 + 1 + 1 = 3
parse_dir(3) -> BUY

SH600001:
0 + 0 + 0 = 0
parse_dir(0) -> SELL
```

Result:

```text
trade_dir:
SH600000      BUY
SH600001      SELL
```

---

### Result After `_agg_order_trade_info()`

```text
              inner_amount   deal_amount   trade_price   trade_value   trade_cost   trade_dir
SH600000      1000           1000          10.10         10100         6            BUY
SH600001      -500           -300          19.90         -5970         6            SELL
```

At this point:

```text
amount is still missing.
```

---

## 5.2 `_update_trade_amount()`

### Purpose

This function adds the outer original target amount:

```text
amount
```

It only uses:

```text
outer_trade_decision
```

It does not depend on the inner execution result.

---

### Example

Assume the outer strategy originally wanted:

```text
Buy SH600000 1200 shares
Sell SH600001 500 shares
```

Then:

```text
amount:
SH600000      1200
SH600001      -500
```

Result after `_update_trade_amount()`:

```text
              amount   inner_amount   deal_amount   trade_price   trade_value   trade_cost   trade_dir
SH600000      1200     1000           1000          10.10         10100         6            BUY
SH600001      -500     -500           -300          19.90         -5970         6            SELL
```

---

### `amount` vs `inner_amount` vs `deal_amount`

These three fields must not be mixed.

```text
amount
    Original target amount from the outer strategy.

inner_amount
    Total target amount generated by the inner strategy after slicing.

deal_amount
    Real executed amount.
```

Example:

```text
Outer strategy wants to buy 1200 shares.
Inner strategy only slices 1000 shares.
The market executes 800 shares.
```

Then:

```text
amount       = 1200
inner_amount = 1000
deal_amount  = 800
```

---

## 5.3 `_update_order_fulfill_rate()`

### Purpose

This function calculates:

```text
ffr = deal_amount / amount
```

It depends on:

```text
deal_amount
    from _agg_order_trade_info()

amount
    from _update_trade_amount()
```

It does not use `inner_amount`.

This means Qlib's `ffr` is calculated from the outer strategy's point of view.

---

### Example

Current data:

```text
amount:
SH600000      1200
SH600001      -500

deal_amount:
SH600000      1000
SH600001      -300
```

Calculation:

```text
SH600000:
ffr = 1000 / 1200 = 0.8333

SH600001:
ffr = -300 / -500 = 0.6
```

Result:

```text
ffr:
SH600000      0.8333
SH600001      0.6
```

Result after `_update_order_fulfill_rate()`:

```text
              amount   inner_amount   deal_amount   trade_price   trade_value   trade_cost   trade_dir   ffr
SH600000      1200     1000           1000          10.10         10100         6            BUY         0.8333
SH600001      -500     -500           -300          19.90         -5970         6            SELL        0.6
```

---

## 5.4 `_agg_base_price()`

### Purpose

This function calculates:

```text
base_price
base_volume
```

`base_price` is the market baseline price used for price advantage.

It is not the real executed price.

The real executed price is:

```text
trade_price
```

and it has already been calculated by `_agg_order_trade_info()`.

---

### Inputs

`_agg_base_price()` depends on:

```text
inner_order_indicators
    Read base_price and base_volume from each inner step.

decision_list
    Used when an inner step does not already have base_price.
    It provides the inner decision and time range.

trade_exchange
    Used to fetch market price and volume if base_price is missing.

pa_config
    Controls TWAP or VWAP.

trade_dir
    Produced by _agg_order_trade_info().
    Used when fetching deal price from the exchange.
```

---

### Inner base_price/base_volume Example

Assume inner indicators contain:

### step0

```text
base_price:
SH600000      10.00
SH600001      20.00

base_volume:
SH600000      1
SH600001      1
```

### step1

```text
base_price:
SH600000      10.10
SH600001      19.90

base_volume:
SH600000      1
SH600001      1
```

### step2

```text
base_price:
SH600000      10.20
SH600001      19.80

base_volume:
SH600000      1
SH600001      1
```

After concatenating along `axis=1`, Qlib gets two matrices:

```text
bp_all_multi_data:

              step0     step1     step2
SH600000      10.00     10.10     10.20
SH600001      20.00     19.90     19.80
```

```text
bv_all_multi_data:

              step0     step1     step2
SH600000        1         1         1
SH600001        1         1         1
```

Here:

```text
rows    = instruments
columns = inner steps
```

---

### Calculating `base_volume`

```text
base_volume = bv_all_multi_data.sum(axis=1)
```

`axis=1` means row-wise summation across inner steps.

```text
SH600000:
1 + 1 + 1 = 3

SH600001:
1 + 1 + 1 = 3
```

Result:

```text
base_volume:
SH600000      3
SH600001      3
```

---

### Calculating `base_price`

```text
base_price = sum(base_price * base_volume) / sum(base_volume)
```

For `SH600000`:

```text
base_price
= (10.00 * 1 + 10.10 * 1 + 10.20 * 1) / 3
= 10.10
```

For `SH600001`:

```text
base_price
= (20.00 * 1 + 19.90 * 1 + 19.80 * 1) / 3
= 19.90
```

Result:

```text
base_price:
SH600000      10.10
SH600001      19.90
```

Result after `_agg_base_price()`:

```text
              amount   inner_amount   deal_amount   trade_price   base_price   base_volume   ffr
SH600000      1200     1000           1000          10.10         10.10        3             0.8333
SH600001      -500     -500           -300          19.90         19.90        3             0.6
```

---

## 5.5 `_get_base_vol_pri()`

### Purpose

This helper is called inside `_agg_base_price()` only when an inner step is missing `base_price`.

Example:

```text
base_price:
SH600000      10.00
SH600001      NaN
```

For `SH600001`, Qlib calls:

```text
_get_base_vol_pri()
```

to fetch market data from `trade_exchange` and calculate:

```text
base_price
base_volume
```

---

### TWAP Mode

Default configuration:

```text
pa_config = {}
agg = "twap"
```

Assume market price series:

```text
time      price
09:30     10.00
09:31     10.10
09:32     10.20
```

TWAP uses equal weights:

```text
base_volume = 1 + 1 + 1 = 3

base_price
= (10.00 * 1 + 10.10 * 1 + 10.20 * 1) / 3
= 10.10
```

---

### VWAP Mode

If:

```text
pa_config = {"agg": "vwap"}
```

Assume market data:

```text
time      price     volume
09:30     10.00     1000
09:31     10.10     3000
09:32     10.20     6000
```

VWAP uses market volume as weights:

```text
base_volume = 1000 + 3000 + 6000 = 10000

base_price
= (10.00 * 1000 + 10.10 * 3000 + 10.20 * 6000) / 10000
= 10.15
```

---

## 5.6 `_agg_order_price_advantage()`

### Purpose

This function calculates:

```text
pa
```

It depends on:

```text
trade_dir
    from _agg_order_trade_info()

trade_price
    from _agg_order_trade_info()

base_price
    from _agg_base_price()
```

Formula:

```text
sign = 1 - trade_dir * 2
pa = sign * (trade_price / base_price - 1)
```

Since:

```text
BUY  = 1
SELL = 0
```

then:

```text
BUY:
sign = 1 - 1 * 2 = -1

SELL:
sign = 1 - 0 * 2 = 1
```

---

### Buy Example

```text
trade_dir   = BUY
trade_price = 10.05
base_price  = 10.10
```

Calculation:

```text
pa = -1 * (10.05 / 10.10 - 1)
   = 0.00495
```

The buy price is lower than the baseline price, so `pa` is positive.

---

### Sell Example

```text
trade_dir   = SELL
trade_price = 20.10
base_price  = 19.90
```

Calculation:

```text
pa = 1 * (20.10 / 19.90 - 1)
   = 0.01005
```

The sell price is higher than the baseline price, so `pa` is positive.

---

# 6. `cal_trade_indicators()`

The previous steps build:

```text
order_indicator
```

`cal_trade_indicators()` aggregates `order_indicator` into:

```text
trade_indicator
```

Example final `order_indicator`:

```text
              amount   inner_amount   deal_amount   trade_price   trade_value   trade_cost   trade_dir   ffr      base_price   base_volume   pa
SH600000      1200     1000           1000          10.05         10050         6            BUY         0.8333   10.10        3             0.00495
SH600001      -500     -500           -300          20.10         -6030         6            SELL        0.6      19.90        3             0.01005
```

---

## 6.1 Aggregated `ffr`

Uses:

```text
order_indicator["ffr"]
```

Default method is mean:

```text
ffr = mean([0.8333, 0.6])
    = 0.71665
```

---

## 6.2 Aggregated `pa`

Uses:

```text
order_indicator["pa"]
```

Default method is mean:

```text
pa = mean([0.00495, 0.01005])
   = 0.00750
```

---

## 6.3 Aggregated `pos`

Uses:

```text
order_indicator["pa"]
```

Formula:

```text
pos = count(pa > 0) / count(valid pa)
```

In this example:

```text
pos = 2 / 2 = 1.0
```

---

## 6.4 Aggregated `deal_amount`

Uses:

```text
order_indicator["deal_amount"]
```

Absolute values are summed:

```text
deal_amount = abs(1000) + abs(-300)
            = 1300
```

---

## 6.5 Aggregated `value`

Uses:

```text
order_indicator["trade_value"]
```

Absolute values are summed:

```text
value = abs(10050) + abs(-6030)
      = 16080
```

---

## 6.6 Aggregated `count`

Uses:

```text
order_indicator["amount"]
```

It counts outer-level orders:

```text
count = 2
```

---

# 7. `record()`

After `cal_trade_indicators()`, Qlib calls:

```python
self.indicator.record(trade_start_time)
```

This saves current metrics into history:

```text
order_indicator_his[trade_start_time] = current order_indicator
trade_indicator_his[trade_start_time] = current trade_indicator
```

These histories are later used to generate indicator DataFrames.

        inner_amount
        deal_amount
        trade_price
        trade_value
        trade_cost
        trade_dir
        base_price
        base_volume

outer_trade_decision
    Used by:
        _update_trade_amount()

    Provides:
        amount

decision_list
    Used by:
        _agg_base_price()

    Provides:
        inner decision and time range for each inner step.
        It is used when base_price must be recalculated.

trade_exchange
    Used by:
        _agg_base_price()

    Provides:
        market price and market volume for baseline price calculation.

# 10. Short Version

```text
atomic=True:
    The executor trades directly.
    Indicator uses trade_info to build order_indicator.

atomic=False:
    The executor is an outer nested executor.
    It does not trade directly.
    It aggregates inner_order_indicators into one outer order_indicator.

The atomic=False flow is:

1. _agg_order_trade_info()
    Aggregates real execution results from inner steps:
    inner_amount / deal_amount / trade_price / trade_value / trade_cost / trade_dir

2. _update_trade_amount()
    Adds outer original target amount:
    amount

3. _update_order_fulfill_rate()
    Calculates:
    ffr = deal_amount / amount

4. _agg_base_price()
    Aggregates baseline market prices:
    base_price / base_volume

5. _agg_order_price_advantage()
    Calculates:
    pa = sign * (trade_price / base_price - 1)

6. cal_trade_indicators()
    Aggregates stock-level order_indicator into step-level trade_indicator.

7. record()
    Saves current order_indicator and trade_indicator into history.
```
